Суть — поднять платформу инференса на Kubernetes, которая обслуживает две принципиально разные модели, потому что это покрывает оба «мира» из вакансии:

LLM (например, Qwen2.5-0.5B/3B или Llama-3.2-1B) → сервится через vLLM или TGI (continuous batching, PagedAttention).
«Классическая» модель — эмбеддер или картиночный классификатор (ResNet/CLIP) → проходит цепочку PyTorch → ONNX → TensorRT и сервится через Triton Inference Server (dynamic batching, model ensemble).

Почему именно две: вакансия упоминает и vLLM/TGI, и Triton/ONNX/TensorRT. Это разные инструменты под разные задачи, и на интервью точно спросят «когда что». Имея в проекте оба, ты отвечаешь из опыта, а не из статьи.
Сверху — слой оркестрации (KServe или Ray Serve), API-гейтвей и асинхронный путь через очередь. Дальше разбито на фазы, можно делать инкрементально.
Фаза 0 — про железо (важно решить сразу)
vLLM и TensorRT требуют NVIDIA GPU. Варианты: локальная карта; аренда (runpod, vast.ai, lambda, или GPU-ноды в GKE/EKS). Прагматичный путь: весь Kubernetes-слой и observability крутишь локально на kind/k3d/minikube с CPU-моделями, а GPU арендуешь только под бенчмарки инференса на несколько часов. Так дешевле и быстрее итерируешь.
Фаза 1 — Baseline serving
Оборачиваешь обе модели в сервис с REST (FastAPI) и gRPC интерфейсом. Тут же отрабатываешь проектирование API: версионирование, схемы запросов, health/readiness-эндпоинты.
Фаза 2 — Containerization + Kubernetes
Docker-образы (multi-stage, минимизация размера), затем Deployment, Service, HPA, resource requests/limits, для GPU — nvidia.com/gpu в limits и device plugin. Сделай HPA не только по CPU, а по кастомной метрике (например, длина очереди или p95-латентность через Prometheus Adapter) — это сильный сигнал на интервью, потому что HPA по CPU для GPU-инференса почти бесполезен.
Фаза 3 — Serving-фреймворки и оптимизация
Здесь главная инженерная мякоть:

LLM на vLLM/TGI с continuous batching, замеряешь throughput vs latency при разных нагрузках.
Маленькую модель прогоняешь через ONNX Runtime, потом строишь TensorRT-движок (FP16, опционально INT8 с калибровкой), сервишь через Triton с dynamic batching.
Профилируешь PyTorch Profiler'ом и nsys (Nsight Systems): находишь узкие места (data loading? препроцессинг на CPU? копирование host↔device?), фиксируешь до/после в цифрах.

Цель фазы — таблица «было/стало»: latency p50/p95/p99, throughput (RPS / tokens/s), GPU utilization, память. Эти цифры — твой главный аргумент на собеседовании.
Фаза 4 — Reliability & distributed patterns

Очередь сообщений (NATS / RabbitMQ / Kafka) для асинхронного инференса: клиент кладёт задачу, воркеры разгребают, результат — через callback или polling. Это даёт backpressure и сглаживание пиков.

API-гейтвей (Envoy / Kong / Istio): rate limiting, маршрутизация, retry-политики.

Идемпотентность: idempotency-key на запись, дедупликация (важно при at-least-once доставке из очереди).
Retry с экспоненциальным backoff + jitter, таймаут-бюджеты, circuit breaker — и обязательно проговори, почему наивные ретраи усиливают аварию (retry storm).

Graceful degradation: фолбэк на кэш / на модель поменьше / отдача «деградированного» ответа + load shedding (сброс лишних запросов вместо падения). Это прямо из вакансии и его часто недоделывают.

Кэширование: Redis для повторяющихся запросов; для LLM — семантический кэш по эмбеддингу запроса.

Фаза 5 — Observability

Prometheus + Grafana: дашборд с латентностью (гистограммы → перцентили), error rate, RPS, и GPU-метрики через DCGM exporter (utilization, memory, температура).
Логи: ELK или Loki + структурированное логирование с trace-id (сквозной через гейтвей → очередь → воркер).
Alertmanager: алерты на p99 > порога, error rate, насыщение GPU-памяти, рост глубины очереди.
Бонус — distributed tracing (OpenTelemetry + Tempo/Jaeger), чтобы видеть, где теряется время в цепочке.

Фаза 6 — Нагрузочное тестирование и хаос
Гоняешь нагрузку через k6 / locust / vegeta (для gRPC — ghz), строишь профиль до насыщения. Потом «ломаешь»: убиваешь под, отрезаешь зависимость, переполняешь очередь — и показываешь на дашборде, что система деградирует контролируемо, а не падает. Это доказательство отказоустойчивости, а не слова.
